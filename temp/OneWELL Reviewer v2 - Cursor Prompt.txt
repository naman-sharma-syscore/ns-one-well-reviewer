# Build: OneWELL Reviewer App (v2) — Vanilla HTML/CSS/JS

You are building a high-fidelity, fully interactive prototype of the **OneWELL Reviewer App** — the
reviewer-side workspace for the WELL Building certification platform (IWBI). This is a companion to a
customer-facing app: where customers complete tasks, reviewers examine submitted evidence and issue
permanent, legally-auditable rulings.

**Core philosophy — "Every review is a Mini-Trial."** A reviewer examines specific evidence, frozen at a
specific moment in time, covering a specific set of locations, and issues a permanent ruling based ONLY on
what was submitted. No spreadsheets, no exports, no legacy DAM complexity. Everything happens online.

## HARD CONSTRAINT: No framework
- Plain **HTML, CSS, and vanilla JavaScript only**. No React, Vue, Svelte, no build step, no bundler, no npm.
- It must run by opening `index.html` directly in a browser (file://) with zero tooling.
- Use **ES modules** (`<script type="module">`) to split code into readable files. Keep each file focused.
- For DOM rendering, write small, explicit render functions that build elements (via `document.createElement`
  or tagged-template -> innerHTML helpers). Do NOT pull in a templating library. A tiny ~15-line `h()` or
  `html``` helper of your own is fine and encouraged for readability.
- State management: a single plain-object store module with a `subscribe(fn)` / `setState(patch)` pattern and
  manual re-render of the active view. No state libraries.
- Routing: **hash-based** (`window.location.hash` + `hashchange` listener). No router library.

/styles/tokens.css        # design-system color + type + spacing tokens (see below)

### Color rules (strict)
- Cyan = brand primary. Use the gradient CTA **exactly once per view** (the single most important action).
- Blue = utility buttons & links. Don't mix cyan AND blue as primaries in the same view.
- Coral = destructive only (fail, SLA breach, errors). Never decorative.
- Emerald = success/done. Plum = medium signal / MRC loop / system notes.
- Concept colors ONLY tag what concept a feature belongs to (chips/dots). Never on buttons, nav, or panels.

### Component primitives (`.ow-*` classes in components.css)
- `.ow-btn` base (inline-flex, center, gap 8px, line-height 1, 6px radius for rects).
  - `.ow-btn-primary` (48px pill, `--gradient-cta-cyan`, white, 16px/600) — one per view.
  - `.ow-btn-solid` (40px, `--blue-600`, white) — Save/Submit/Continue.
  - `.ow-btn-outline` (40px, white, inset 1px `--gray-200` border) — Cancel/Back.
  - `.ow-btn-ghost`, `.ow-btn-icon` (32px square).
  - Size variants `--sm` (28px) / `--md` (36px) / `--lg` (48px).
- `.ow-status` — 24px uppercase pill, 12px/600, for state.
- `.ow-concept-tag` — 24px square-corner (4px) uppercase chip, color-coded by concept.
- `.ow-code-chip` — 22px, 11px/600, tabular-nums, for feature codes (W01, A02), concept-colored bg.
- Inputs: white bg, 1px `--gray-200`, 6px radius, 40px height; focus = 2px `--cyan-300` ring + `--cyan-50` halo.
- Labels: 12px `--gray-600`/500, sentence case, above the field.

### Voice
Calm, factual, generous, never alarmist. Sentence case headings. "8 points to Gold" not "You're failing".
Errors are `<what happened> + <next step>`, never bare "Error".

### Layout shell

## DATA ENGINE (`js/data.js`) — single source of truth

Anchor "now" deterministically: `const NOW = new Date('2026-06-01T14:30:00Z')` so SLA math is stable across
reloads. Provide helpers: `fmt(date)`, `relative(date)` ("3d ago" / "in 5h"), `slaBucket(due)` ->
`{kind:'ok'|'soon'|'overdue', label, hours}` (soon = within 48h, overdue = past due), and
`sampleSize(n) = Math.ceil(Math.sqrt(n))`.

### Model sequence
`Target/Strategy -> Task -> ReviewSnapshot -> Notes/Rulings -> Achievements`

### Concepts (10) and Features (real WELL v2 codes)
Concepts with their tag color + readable text color: Air, Water, Nourishment, Light, Movement, Thermal Comfort,
Sound, Materials, Mind, Community.
Features (at minimum): **A01** Air quality (pre), **A02** Smoke-free environment (pre), A05 Enhanced ventilation,
A07 Operable windows, **W01** Water quality indicators (pre), **W02** Drinking water quality (pre),
**W03** Basic water management (pre), W04 Enhanced water quality, **T01** Thermal performance (pre),
T03 Thermal zoning, T05 Humidity control, L03 Circadian lighting, **N01** Fruits & vegetables (pre),
V02 Ergonomic workstations, S02 Maximum noise levels, X06 VOC restrictions, M07 Restorative spaces,
C04 Accessibility & universal design. ("pre" = precondition flag.)

### Reviewers (>=3; current session = first)
- rev-001 **Maya Reyes** — Senior WELL Reviewer · Air & Water (isMe: true), cohorts [Air, Water, Thermal].
- rev-002 Jordan Tan — Materials & Mind. rev-003 Adaeze Balogun — Lead · Performance.

### Accounts/Portfolios with location counts
Helix Realty (50 locations), Northbay Asset Mgmt (16), Ridgeway Hospitality (12), Meridian Logistics (1 site),
Keystone Health Network (8), Aria Workplaces (22). Give each named reference locations with real cities
(Chicago, Singapore, Tokyo, Austin, Boston, Berlin, etc.).

### Three verification paths
- **Shareable** — ONE artifact covers an entire portfolio (e.g., Helix Smoke-Free Policy A02 covering all 50
  locations; one ruling applies to all).
- **Audited** — `ceil(sqrt(N))` sampling. e.g., Northbay W01 over 16 locations -> 4 sampled tasks; ruling extends
  to all 16. Ridgeway 12 -> 4 sampled.
- **Per-location (Individual)** — each site uploads its own evidence (e.g., Keystone W02 per-clinic;
  Meridian A01 single warehouse).

### Tasks — seed 25+ across ALL statuses and ALL paths
Status vocabulary with pill colors:
- `READY` "Ready for review" — cyan-100/cyan-800
- `UNDER` "Under review" — blue-100/blue-700
- `ATTENTION` "Needs attention" (returned to client / in MRC loop) — plum-100/plum-700
- `COMPLETED` "Completed" — emerald-100/emerald-700
- `FAILED` "Failed" — coral-100/coral-700

Each Task carries: id (`task-XXXX`), accountId, feature, concept, path, status, `locationIds[]`,
`sampledLocationIds` (audited only), portfolioSize, assignedReviewerId, submittedAt, dueAt, computed `sla`,
`snapshot`, `ruling`, `notes[]`, `mrcCycle` (0-3), points.

A **ReviewSnapshot** = the frozen evidence package: `{capturedAt, submittedBy, attachments[], narrative}`.
Attachments use realistic filenames: `HVAC_Assessment_Chicago_Q1_2026.pdf`,
`Water_Testing_Results_Singapore.pdf`, `Helix_Smoke_Free_Policy_v3.2.pdf`, with type (pdf/archive/sheet/data)
and human size ("482 KB", "14.2 MB").

### WELL v2 Verification Checklists per feature
Map each feature code -> array of criteria `{id, tier:'T1'|'T2', text}`. T1 = required, T2 = bonus. Provide real
thresholds, e.g. W02: "Point-of-use filtration installed at all drinking water outlets", "Filter maintenance log
within manufacturer interval", "Quarterly samples in last 12 months" (T1); "Annual third-party verification"
(T2). W01 includes "Lead <= 0.005 mg/L", "Total coliform absent". A02 includes "Written smoke-free policy",
"Covers e-cigarettes", "Signage at all entries". Provide a GENERIC_CHECKLIST fallback for features without a
curated list.

### Saved-view filters & rollup
Views: `all` (All assigned), `urgent` (SLA at risk & not closed), `progress` (Under review), `mrc` (Needs
attention), `history` (Completed or Failed). Each a pure predicate `(task, meId) => bool`.
`rollup(tasks, meId)` -> `{assigned, sla, awaiting, doneWeek}`.

## STATE MACHINE & AUTOMATION (implement in store + view-task.js)
- **Locking** — the first edit on a task (checking a criterion or flagging a doc) fires a simulated
  `TaskService::lock()`: show a plum "Customer workspace locked" banner and transition status READY -> UNDER.
- **Auto-generated system notes** — every status transition injects a stylized SYSTEM note into the ledger:
  `state(READY) -> state(UNDER) · reason="first reviewer edit" · by=rev-001`. MRC send adds a public note;
  approve/fail add public notes.
- **Achievement generation** — clicking **Complete · Approve** transitions to COMPLETED, injects an
  `AchievementService::generate(...)` system note, and opens a **success modal** showing the immutable
  Achievement record as pretty-printed JSON (achievement_id, task_id, feature, decision, ruled_at, reviewer,
  scope {path, portfolio_size, sampled}, tier_score {tier_1, tier_2 verified counts}, snapshot_sha, immutable:true).
- **Shared store** — tasks live in ONE store. A ruling issued in the task view must immediately update the
  dashboard metrics and the ops analytics. Re-derive SLA buckets whenever tasks change.
